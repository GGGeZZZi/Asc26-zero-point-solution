# ASC26 参赛日志 – AMSS‑NCKU 优化之路（个人视角）

## 📅 时间线
- **2025.11** 报名 ASC26，确认双赛道（任务 A + 任务 B）  
- **2025.12 – 2026.03** 四个多月的持续攻关，最终提交结果  

## 🧑‍💻 角色定位
我是团队中 **AMSS‑NCKU（任务 B）数值相对论代码** 的主要负责人，环境搭建、代码编译、性能测试、方案探索及提案撰写全部由我独立完成。

## 🌱 初识题目：一脸懵逼到慢慢入门
- 拿到题目时完全不知何是爱因斯坦场方程、自适应网格细化、MPI 并行。  
- 先通读官方文档，反复研读 CSAMR 论文（2016），起初看伪代码和公式头大。  
- 通过 **“边看论文、边对照代码”** 的方式，逐渐理清函数与循环对应关系，代码结构在脑中变得清晰。

## 🛠️ 环境搭建：踩坑的开始
- 学校超算系统较老旧，GCC 仅 4.8.5，许多现代库不支持。  
- 需要 **静态编译 Python 3.8.18**，然后安装 numpy、matplotlib 等依赖。  
- 因超算不可外网，我在个人虚拟机下载好所有包，再用 `scp` 传至超算——每次传输耗时且易断连。  
- 安装过程中不断遇到 **libgfortran 版本不匹配**、MATLAB 库路径污染环境变量等错误，一度想放弃。  
- 经过 **卸载冲突库、手动设置 `LD_LIBRARY_PATH`、使用 `rpath` 把库路径写进可执行文件** 等多种手段，整整两天后终于能够顺利编译并运行。  
- 这段经历让我对 Linux 库管理、编译链接有了深刻体会，虽痛苦但算是因祸得福。

## ⚙️ 性能测试：发现 16 核的秘密
- 完成基准测试后，4 核跑 10 秒物理时间竟需 **120 分钟** —— 初始化部分（串行 TwoPuncture）占用约 30 分钟，说明后续优化空间巨大。  
- 系统测试不同核心数：4 → 8 → 12 → 16 → 24 核，每次修改 `MPI_processes` 与 PBS 脚本的 `ppn`，提交作业并等待结果。  
- **惊喜发现**：16 核的表现竟优于 24 核！多次重复测试均得此结论。  
- 文献查证后明白：**内存带宽饱和导致通信开销过大**，核心数不是越多越好——这让我对并行计算的“甜 spot”有了切身感受。

## 🚀 编译优化：-flto 带来的惊喜
- 在确定 16 核为最佳配置后，先尝试 `-O3`，性能不仅未提升反而略降，信心受挫。  
- 偶然看到 `-flto`（链接时优化）在社区里有好评，决定试一试。  
- **仅添加 `-flto` 一个编译选项**，性能竟提升 **约 7%**：  
  - 10 秒模拟从 **50.33 分钟** 降至 **46.8 分钟**。  
- 这一瞬间让我觉得之前的所有努力都没有白费。  
- 后续又尝试了 `-Os`、MKL 数学库、PGO 反馈优化，均未再有突破；MKL 甚至导致性能倒退。这些失败的尝试均被详细记录，后来写入提案，展示我们的探索过程。

## 🧪 CSAMR 尝试：一次“失败”但值得的冒险
- CSAMR（Concurrent Adaptive Mesh Refinement）是 AMSS‑NCKU 开发者提出的新型并行方法，理论上可大幅提升并行效率。  
- 我花费约一周时间，仔细研究论文伪代码，在 `bssn_class.C` 中定位对应的 `RecursiveStep` 函数，按照论文思路将原递归调用改为 **所有层同时演化**，并加上限制与延拓操作。  
- 编译通过后，满怀期待跑了一秒测试——**结果竟然只推进了 0.7 秒/小时**，远慢于基线的 1 秒/小时。  
- 初时感到崩溃，认为这一周全部白费。冷静下来后逐步分析：  
  - CSAMR 要求 **所有层都使用最细层的步长**，导致粗网格的计算量呈指数级增长。  
  - 在只有 16 核的环境下，这一副作用完全抵消了并行带来的收益。  
- 这一“失败”其实**验证了论文的前提**：CSAMR 需要大量核心才能发挥优势。在此过程中，我对 AMR 算法的理解有了质的飞跃。

## 🏁 最终模拟：32.8 小时的等待
- 采用 **16 核 + `-flto`** 配置提交了 1000 秒的正式作业。  
- 提交后每天刷新 `qstat`，观察进度——那种等待感就像考试后等成绩一样煎熬。  
- 两天后任务完成，输出显示 **总耗时 118170 秒 ≈ 32.8 小时**，比我最初预估的 45.6 小时还快了不少。  
- 检查约束图和轨迹图均符合要求（约束违反 ≤ 2，轨迹 RMS ≤ 1%），那一刻觉得所有的辛苦都值得了。
